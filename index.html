02 · How it works

The board is a byproduct of building.

Your agent reads and writes files while it works. Boards just points that habit at a markdown file — so keeping the roadmap current costs nothing extra.

The file is the database

A board is any markdown file you name — PROJECT.md, MARKETING.md, RELEASES.md, whatever stream of work you want to track — checked into the repo. No schema, no migration, no SaaS. Want the history? Run git log.

Atomic with the code

The agent edits the board as it builds, so the board diff lands in the same commit as the code it describes. No sync job, no second surface to rot — the commit is the proof.

~70 lines, zero deps

A tiny parser — no markdown library — renders the file inside your own admin, or as a standalone page, in whatever environment you choose. It's all text: no dangerouslySetInnerHTML, no XSS surface, no supply chain.

03 · The format contract

Plain text a human and an agent both read.

Talk to your agent in plain language — "add a bug for the export timeout", "move T-12 to done", "what's left before launch?" — and it edits the markdown to match. The whole grammar fits on a napkin.

04 · The honest part

A convention, not a tool.

An agent keeping tasks in sync isn't a new idea, and there's genuinely good software here. Backlog.md is a git-native task CLI built for agents — its board is committed markdown too; you just drive it through a command you invoke ("no automatic synchronization"), by design. claude-todos runs a self-managing todo backend beside your code. Others expose a kanban board to agents over MCP. If you want a hosted product with a write-UI, one of those is the right call — and we'd point you to it.

Boards makes a different trade. There's nothing to invoke and nothing to run: no CLI, no MCP server, no database, no service. The board is just a file your agent already edits, and the only "app" is a small renderer you drop into your own admin — or a standalone page. You give up a write-UI and a hosted backend. In return, the board is the repo — versioned, diffable, and updated in the same commit as the code that moved it.

Reach for a tool Backlog.md · claude-todos · MCP boards

  • → You want a write-UI or a hosted board others can click.
  • → You want a CLI or MCP command to invoke and script against.
  • → You want a separate app or backend others can use without the repo.

Reach for Boards this page

  • ✓ You want the board to be the repo — versioned and diffable.
  • ✓ You want zero runtime, zero deps, zero services to operate.
  • ✓ You want the board to move in the same commit as the code.
